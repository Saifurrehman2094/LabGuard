# Impeccable — LAB-Guard

## Design Context

### Users

- **Scope**: University coursework project; not for commercialization. If deployed, **only on the university’s lab computers**.
- **Contexts**: Controlled computer-lab exams on Windows (Electron desktop).
- **Roles**: Administrators (courses, configuration), teachers (exams, evaluation, monitoring, AI-assisted workflows), students (exams, submissions, face enrollment where used).
- **Emotional goals**: **Students** should feel **calm**. **Teachers** should feel the system is **fair** and **trusted**. High-stakes exam context — reduce unnecessary alarm; monitoring surfaces stay factual and legible.

### Use cases

- Course and roster management; full exam lifecycle (create, publish, edit, take, submit).
- Proctoring: desktop monitoring, optional screenshots, camera pipeline and violations.
- Programming assessment: C++ evaluation, weighted cases, teacher analytics.
- Optional LLM flows when API keys are configured.

### Brand personality

- **Stated anchor**: *Elegant* — executed as refined layout and type, not fussy ornament.
- **Blend**: **Playful** touches inside an **institutional** frame — credible lab software with moments of warmth and personality (never flippant about integrity or grades).
- **No fixed external references** or anti-references supplied; direction is internal to this brief.

### Aesthetic direction

- **Theme**: **Light mode only** (no dark-default requirement).
- **Palette**: No mandatory brand colors; choose a cohesive light palette that feels calm and slightly playful (tinted neutrals, restrained accent), avoiding generic “AI dashboard” cyan-purple neon tropes.
- **Engagement**: UI/UX should feel **engaging and crafted**, not boring — captivate through **hierarchy, typography, spacing, and purposeful motion**, not clutter or gimmicks.

### Design principles

1. **Calm student surfaces**: Exam flows stay focused; generous rhythm; clear primary actions; avoid visual panic on routine states.
2. **Fair, trusted teacher tools**: Monitoring and grading views prioritize **clarity, timestamps, and scannable structure** so decisions feel evidence-based.
3. **Playful × institutional**: Small delights (micro-motion, illustration or shape language, friendly copy where appropriate) **within** a serious academic shell.
4. **Light-first craft**: Polish lives in type scale, color discipline, and spacing — OKLCH-friendly tokens over time; respect `prefers-reduced-motion`.
5. **Engaging ≠ loud**: Interest comes from intentional contrast and composition; do not sacrifice readability or trust for decoration.

### Implementation baseline (technical)

- React 19 + TypeScript, Electron; tokens in `frontend/src/styles/design-system.css` (today: blue primary, emerald secondary, system fonts). Evolve tokens and components to match this context; avoid one-off styles that fight the system.
